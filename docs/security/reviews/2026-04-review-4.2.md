2. Tighten the documented contract around production metrics exposure, especially the intentionally public override.

I did not run the full test suite from the concatenated artifact. This review is based on static inspection of the attached bundle, implementation snippets, tests included in the bundle, and the contract docs.

# 1. OpenAI Chat Completions passthrough and explicit unsupported behavior

## Status: resolved, with one important follow-up

The previous schema blocker is fixed. `ChatCompletionRequest` now uses Pydantic `ConfigDict(extra="allow")`, while still requiring `model` and `messages`. That means ordinary SDK fields no longer disappear at the API boundary. `slaif-api-gateway-concatenated.txt`

The OpenAI compatibility doc now explicitly states that the gateway preserves common Chat Completions fields including `temperature`, `top_p`, `stop`, `tools`, `tool_choice`, `response_format`, `seed`, `user`, `logprobs`, `top_logprobs`, penalties, `n`, `stream_options`, `metadata`, `reasoning_effort`, `modalities`, `parallel_tool_calls`, `service_tier`, and token controls. It also says unknown ordinary JSON-compatible fields are preserved unless policy explicitly rejects them. `slaif-api-gateway-concatenated.txt`

The compatibility contract is also clear that unsupported `/v1/responses`, `/v1/embeddings`, files, images, audio, and native Anthropic endpoints are not implemented, and unsupported `/v1` routes return OpenAI-shaped errors through FastAPI error handlers. `slaif-api-gateway-concatenated.txt`

### Remaining issue: `n` must be policy-handled

The docs say `n` is preserved, but the described request-policy rejection list does not mention rejecting `n > 1` or multiplying quota reservation by `n`. `slaif-api-gateway-concatenated.txt`

That is a real accounting edge case. If a user sends `n=3`, completion tokens and cost can multiply while the reservation may still be based on a single output budget unless the code handles this elsewhere. Before feature expansion, either:

- reject `n` values other than `1`,
- multiply output-token and cost reservation by `n`,
- or document `n > 1` as unsupported.

This is the most important remaining policy issue I found.

# 2. `stream_options.include_usage` enforcement

## Status: resolved

The gateway now has a dedicated helper that forces streaming usage options: it copies the upstream body, sets `stream=True`, and ensures `stream_options.include_usage=True`, preserving other `stream_options` keys when present. `slaif-api-gateway-concatenated.txt`

OpenAI adapter coverage verifies that a client-provided `stream_options={"include_usage": False, "other": "preserved"}` is forwarded upstream as `{"include_usage": True, "other": "preserved"}`, with `Accept: text/event-stream`, provider authorization, and upstream model substitution. `slaif-api-gateway-concatenated.txt`

OpenRouter has equivalent coverage: it forces `include_usage=True`, preserves ordinary body fields such as `temperature`, `tools`, `tool_choice`, and `response_format`, and does not mutate the original request body. 

This directly resolves the prior streaming-accounting precondition problem.

# 3. Streaming completed-without-usage behavior

## Status: resolved

The current contract says final provider usage is required for successful streaming accounting. If final usage is missing, the gateway records a failed/incomplete event, releases the reservation according to policy, emits a safe SSE error event, and does **not** emit a normal successful `[DONE]`. `slaif-api-gateway-concatenated.txt`

The implementation snippet matches that contract: in the completed-without-usage branch, it records provider failure/release and yields an OpenAI-shaped SSE error with code `stream_usage_missing`; it does not yield the saved provider `[DONE]` event. `slaif-api-gateway-concatenated.txt`

There is also explicit unit coverage asserting the response contains `stream_usage_missing`, mentions final usage metadata, and does not contain `data: [DONE]`. Another test checks that the error event does not leak streamed content or a secret-like gateway key. `slaif-api-gateway-concatenated.txt`

This is a meaningful improvement over the previous version, where missing usage could look like a normal successful stream.

# 4. Provider-completed accounting finalization recovery and reconciliation

## Status: resolved

This was one of the most important previous blockers, and the new design is much better.

The streaming code now records a durable provider-completed usage record **before** final accounting finalization. If finalization raises `AccountingError` or `QuotaError`, it marks that provider-completed record as finalization-failed and then emits a safe stream error instead of pretending the provider failed or releasing the call as zero-cost. 

The accounting service creates provider-completed rows with recovery metadata and later marks them with `recovery_state`, `needs_reconciliation=True`, and a safe finalization error code if finalization fails. 

There is integration coverage for a simulated finalization failure after streamed content was delivered: the client receives the completion text and an accounting error, `[DONE]` is suppressed, the reservation remains pending with reserved counters intact, used counters remain zero, and the ledger is marked as failed/accounting-finalization-failed rather than as a provider failure. `slaif-api-gateway-concatenated.txt`

The CLI now exposes `quota list-provider-completed-recovery` and `quota reconcile-provider-completed`; reconciliation defaults to dry-run, warns on execution, accepts either usage-ledger or reservation ID, and has safe-output tests. 

This resolves the prior “content delivered but accounting failed” production blocker.

### Remaining operational caveat

The recovery is manual/operator-driven. That is acceptable for the current phase because it is documented, but before a production launch this should become a scheduled maintenance job or at least have alerting around unreconciled provider-completed rows.

# 5. Redis active-concurrency behavior for long streams

## Status: resolved

The previous Redis model treated concurrency as “requests started within the rate window,” which was wrong for long SSE streams. The current docs now describe active request IDs separately from rate windows: streaming responses refresh the Redis active slot while open, release removes the specific request ID, and TTL is a crash-cleanup fallback. `slaif-api-gateway-concatenated.txt`

Settings now validate that the heartbeat interval is less than the concurrency TTL, which is the right basic safety invariant. `slaif-api-gateway-concatenated.txt`

Most importantly, there is integration coverage where a first stream stays open, the test waits past a short interval, a second request is rejected with `concurrency_rate_limit_exceeded`, and a third request succeeds after the first stream finishes. `slaif-api-gateway-concatenated.txt`

That directly addresses the earlier long-stream concurrency bug.

# 6. CLI JSON/plaintext key output safety

## Status: resolved

The CLI secret-output behavior is now substantially safer. Tests show that `keys create --json` refuses to proceed unless the operator explicitly requests secret output; the fake create function is not called, the JSON error is safe, and the plaintext key is absent from stdout. `slaif-api-gateway-concatenated.txt`

The intended exceptional cases are now explicit:

- `--json --show-plaintext` includes the plaintext once and warns on stderr.
- `--json --secret-output-file PATH` writes the secret to a file and excludes it from stdout. `slaif-api-gateway-concatenated.txt`

The same pattern exists for rotation: `keys rotate --json` requires explicit secret output, `--show-plaintext` includes the replacement once with a warning, and `--secret-output-file` keeps stdout secret-free. `slaif-api-gateway-concatenated.txt`

README now matches this behavior, stating that JSON mode is secret-safe by default and that plaintext capture requires `--show-plaintext` or `--secret-output-file PATH`. `slaif-api-gateway-concatenated.txt`

This resolves the previous CLI leakage blocker.

# 7. Custom-prefix and nested metadata redaction

## Status: resolved

The redaction tests now cover configured custom prefixes and generic gateway-key-shaped fallback. A generated `sk-acme-prod-...` key is redacted as `sk-acme-prod-<public_id>.***`, and the tests assert that the secret component and even its first/last fragments are not retained. There is also a fallback test for an unknown `sk-lab-` prefix. `slaif-api-gateway-concatenated.txt`

Nested metadata sanitization now covers camelCase, snake_case, and kebab-case sensitive key variants, including `providerApiKey`, `authorizationHeader`, `accessToken`, `refresh_token`, `tokenHash`, `encryptedPayload`, `secretValue`, `csrfToken`, `sessionCookie`, `openaiKey`, `openrouterKey`, and password-like keys. 

README also reflects this broader redaction model, saying configured prefixes and generic gateway-key-shaped values are recognized and secret characters are not preserved. `slaif-api-gateway-concatenated.txt`

This resolves the prior redaction issues.

# 8. Provider diagnostics and cost metrics

## Status: resolved enough for this phase

Provider diagnostics are now much more professional. The diagnostics builder creates bounded, sanitized provider diagnostics from response bodies and headers, extracts safe upstream request IDs, redacts provider/gateway keys, drops content-bearing fields, and deliberately avoids persisting plain-text provider bodies because they may contain echoed prompts or completions. `slaif-api-gateway-concatenated.txt`

The docs state that raw provider response bodies are not returned to clients or stored, while sanitized diagnostics may be stored for operators. `slaif-api-gateway-concatenated.txt`

The provider-forwarding contract says successful accounting records token and cost metrics, and README says successful accounting finalization records finalized EUR cost in Prometheus metrics. 

I would still add one more direct test that asserts `gateway_cost_eur_total` increments on both non-streaming and streaming finalized calls. The contract claims it; the implementation appears intended to do it; a metric-specific assertion would close the loop.

# 11. Production `/readyz` and `/metrics` exposure controls

## `/readyz`: resolved

Production readiness no longer exposes Alembic revision details by default. Tests verify that production `/readyz` returns only coarse `status`, `database`, `schema`, and `redis` fields, and that revision strings and DB URL secrets are absent. There is an explicit `READYZ_INCLUDE_DETAILS=True` override. `slaif-api-gateway-concatenated.txt`

PostgreSQL integration coverage also verifies production `/readyz` hides schema revisions by default and can include them when explicitly configured. `slaif-api-gateway-concatenated.txt`

## `/metrics`: mostly resolved

Metrics are denied in production by default and allowed only through explicit configuration or allowlisted IPs. Tests cover production default `403`, explicit public override, configured IP allowlist, disabled metrics returning `404`, and absence of secrets in metrics output. `slaif-api-gateway-concatenated.txt`

The docs also warn that `METRICS_PUBLIC_IN_PRODUCTION=true` intentionally makes metrics public and should not be used for internet-facing deployments. `slaif-api-gateway-concatenated.txt`

### Remaining concern

The explicit public override is acceptable for development or controlled private deployments, but it is a footgun for an open-source infrastructure project. I would keep it only if:

- it is named very loudly, as it is now,
- it logs a production startup warning,
- deployment docs recommend never setting it on internet-facing systems,
- and tests verify default production remains closed.

## Status: mostly accurate

The requested contract docs are now substantially aligned with implementation.

`docs/openai-compatibility.md` accurately describes implemented endpoints, unsupported endpoints, request-field preservation, gateway-mutated fields, streaming usage requirements, missing-usage stream errors, provider-completed recovery, Redis heartbeat behavior, and OpenAI-shaped errors.

### Documentation issue to fix

The docs should explicitly define the policy for `n`. Because `n` is listed as preserved, the contract should state whether:

- `n > 1` is rejected,
- quota estimates are multiplied by `n`,
- or the gateway supports `n` only opportunistically without hard accounting guarantees.

Right now, preserving `n` without a visible accounting policy is the main documentation/behavior ambiguity.

# Outstanding issues before next feature expansion

## Must fix before broad feature expansion

### 1. Define and enforce `n` accounting

Because `n` is preserved as an ordinary Chat Completions field, the gateway must either reject `n > 1` or multiply quota reservation and output-cost estimation by `n`. This is now the clearest remaining hard-accounting risk. `slaif-api-gateway-concatenated.txt`

Recommended policy for the next sprint:

```text
If n is absent: treat as 1.
If n == 1: allow.
If n > 1: reject for now with OpenAI-shaped invalid_request_error.
```

That is safer than trying to support multi-choice accounting immediately.

### 2. Add direct metric assertions for finalized cost

The docs and contract say finalized EUR cost is recorded in Prometheus metrics. Add tests that perform one finalized non-streaming request and one finalized streaming request, then assert the cost metric increments with expected provider/model/endpoint labels. 

### 3. Add startup warning for public production metrics

`METRICS_PUBLIC_IN_PRODUCTION=true` is intentionally supported, but it is dangerous. Tests confirm the public override works. `slaif-api-gateway-concatenated.txt`

Add a loud structured startup log warning when this is enabled in production.

## Should fix soon

### 4. Add scheduled reconciliation or alerting

Provider-completed finalization recovery is now manual and operator-repairable. That is a big improvement, but production operators need alerts for unreconciled recovery rows and stale reservations. `slaif-api-gateway-concatenated.txt`

### 5. Add official-client assertion for missing-usage stream error shape

The docs say this can be added later. I agree it is not a blocker anymore, but it would harden the streaming contract for real SDK behavior. `slaif-api-gateway-concatenated.txt`

### 6. Clarify OpenRouter cost semantics in operator docs

The matrix correctly says OpenRouter provider cost metadata is partial and that hard quota finalization still uses configured pricing/FX. `slaif-api-gateway-concatenated.txt`

That distinction should be repeated anywhere operators might assume OpenRouter-reported `usage.cost` is the billing source of truth.
