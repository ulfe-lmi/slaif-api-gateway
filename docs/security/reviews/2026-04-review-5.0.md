# Audit 5.0 verdict

Using only the newly attached Review/Audit 5.0 bundle, I would now rate the project as a **serious pre-production infrastructure system** for the implemented scope: authenticated `/v1/models`, non-streaming `/v1/chat/completions`, streaming `/v1/chat/completions`, OpenAI/OpenRouter forwarding, PostgreSQL quota/accounting, Redis operational throttling, CLI operations, admin web foundation, and explicit key-email delivery.

**Overall grade: B+**

The codebase has moved beyond “credible alpha.” The old central blockers around request passthrough, streaming usage, provider-completed recovery, Redis long-stream concurrency, CLI plaintext JSON leakage, custom-prefix redaction, provider diagnostics, and DB pool/readiness configuration appear substantially resolved. The remaining issues are narrower but still important: **`n > 1` accounting**, **admin login brute-force protection**, **`/v1/models` policy semantics drift**, and **production provider-secret validation drift**.

I extracted and syntax-compiled the Python files from the new bundle successfully. I did not run the full test suite, so this is a static/source-and-docs audit, not a test execution report.

# Previously identified blockers: current status

## 1. OpenAI Chat Completions passthrough

**Status: resolved, with one remaining hard-accounting exception.**

The compatibility docs now say `ChatCompletionRequest` requires only `model` and `messages`, preserves extra JSON-compatible fields, and lists ordinary Chat Completions fields including `temperature`, `top_p`, `stop`, `tools`, `tool_choice`, `response_format`, `seed`, `user`, `logprobs`, penalties, `n`, `stream`, `stream_options`, `metadata`, reasoning/service fields, and token controls. `slaif-api-gateway-concatenated.txt`

The compatibility matrix also states that common fields and unknown ordinary JSON fields are preserved, while output-token controls are validated/defaulted and conflicting values are rejected. `slaif-api-gateway-concatenated.txt`

The remaining exception is `n`. The docs list `n` as preserved, the remediation matrix still marks "`n > 1` Chat Completions accounting policy" as **Planned / Not implemented**, and the cost-estimation code shown in the bundle estimates output cost from `policy.effective_output_tokens` without multiplying by `n`.

**Fix before broad expansion:** reject `n > 1` for now with an OpenAI-shaped `invalid_request_error`, or multiply all output-token and cost reservations by `n`.

---

## 2. `stream_options.include_usage` enforcement

**Status: resolved.**

Request policy now forces `stream_options.include_usage=true` when `stream` is `true`, preserves other stream options, rejects non-object `stream_options`, and injects a default `max_completion_tokens` when no output-token control is present. `slaif-api-gateway-concatenated.txt`

The provider-forwarding contract says both OpenAI and OpenRouter streaming requests use `Accept: text/event-stream`, preserve ordinary body fields, replace only the resolved upstream model, and force usage options for streaming. `slaif-api-gateway-concatenated.txt`

---

## 3. Streaming completed-without-usage behavior

**Status: resolved.**

The docs now state that missing final usage emits a safe stream error, records an incomplete/failed accounting event, does not charge actual cost, and does not emit normal successful `[DONE]`. `slaif-api-gateway-concatenated.txt`

The included PostgreSQL streaming test verifies exactly that behavior: a stream without final usage contains `stream_usage_missing`, mentions final usage metadata, does **not** contain `data: [DONE]`, releases the reservation, clears reserved counters, records failed accounting, and avoids prompt/completion/secret leakage in metadata. `slaif-api-gateway-concatenated.txt`

---

## 4. Provider-completed accounting finalization recovery

**Status: resolved for the current phase.**

The provider-forwarding contract says streaming writes a durable provider-completed record before final counter mutation; if finalization fails, the record is marked `needs_reconciliation=true` with `recovery_state=provider_completed_finalization_failed`, and `[DONE]` is not emitted. `slaif-api-gateway-concatenated.txt`

The security model also states that manual reconciliation exists for expired pending reservations and provider-completed streaming finalization failures, and that provider-completed repair uses stored safe usage/cost metadata without calling providers. `slaif-api-gateway-concatenated.txt`

This is a substantial improvement over earlier versions. The remaining production-hardening item is **automation/alerting**, not core correctness: operators should get alerts for unreconciled provider-completed rows and stale reservations.

---

## 5. Redis active concurrency for long streams

**Status: resolved.**

The OpenAI compatibility docs state that streaming Redis concurrency slots are heartbeated while open and released in generator cleanup. They also state that a real ASGI server test closes a stream early and verifies cleanup. `slaif-api-gateway-concatenated.txt`

The security model describes Redis as operational throttling only and says active concurrency uses request-specific slots with heartbeat refresh for long streams and TTL cleanup for crash recovery. `slaif-api-gateway-concatenated.txt`

This addresses the earlier “long streams age out after the rate window” defect.

## 6. CLI JSON/plaintext key output safety

**Status: resolved.**

The key CLI now refuses JSON plaintext output by default unless the operator explicitly uses `--show-plaintext` or `--secret-output-file`; send-now/enqueue email modes reject additional plaintext destinations, and secret files are written as one-time outputs with warnings. `slaif-api-gateway-concatenated.txt`

The security model matches this behavior: text mode may show a newly generated key once, JSON is secret-safe by default, and `--secret-output-file` writes a new file with `0600` permissions. `slaif-api-gateway-concatenated.txt`

## 7. Custom-prefix and nested metadata redaction

**Status: resolved.**

The security model now states that structured logging redacts configured/custom gateway key prefixes, bearer keys, provider keys, cookies, passwords, CSRF/session tokens, token hashes, encrypted payloads, nonces, and nested sensitive metadata across common key naming styles. `slaif-api-gateway-concatenated.txt`

The README says redaction recognizes configured gateway key prefixes and generic gateway-key-shaped values, and that it does not preserve secret characters from the key secret component. `slaif-api-gateway-concatenated.txt`

# New or still-outstanding issues before next expansion

## P0/P1 - `n > 1` can break hard quota assumptions

This is the clearest remaining accounting risk.

The gateway preserves `n`, but the remediation matrix still says the `n > 1` accounting policy is not implemented. Pricing estimates use the single `policy.effective_output_tokens` value, not `n × effective_output_tokens`.

For OpenAI Chat Completions, `n > 1` can cause multiple completions, multiplying output tokens and cost. If the gateway reserves as though only one completion can be produced, it can under-reserve cost/tokens before the upstream call.

**Recommended immediate policy:**

```text
n absent -> treat as 1
n == 1 -> allow
n > 1 -> reject with OpenAI-shaped invalid_request_error
```

Supporting `n > 1` later is fine, but only after multiplying quota reservation, pricing estimate, and actual-usage validation semantics.

## P1 - `/v1/models` visibility conflicts with documented empty `allowed_models` semantics

The database schema says:

```text
allow_all_models false + empty allowed_models means no models allowed
```

`slaif-api-gateway-concatenated.txt`

But the current model catalog test asserts that `allow_all_models=False` with empty `allowed_models=()` returns all visible models, treating the empty list as “no restriction.” `slaif-api-gateway-concatenated.txt`

The compatibility docs say `/v1/models` returns models visible/allowed for the authenticated key.

This is probably not a direct call-authorization bypass, because route resolution for actual chat requests rejects disallowed models. But it is still a governance and information-disclosure problem: a key configured with no model access can see the model catalog.

**Fix:** make `ModelCatalogService` mirror route authorization:

```text
if allow_all_models is false and allowed_models is empty:
    return []
```

Then update the test that currently asserts the opposite.

---

## P1 - Admin login lacks visible brute-force/rate-limit protection

Admin sessions and CSRF are well designed: login creates server-side sessions, tokens are HMAC-hashed, cookies are secure/HttpOnly/SameSite, and state-changing forms require session CSRF. `slaif-api-gateway-concatenated.txt`

But the admin authentication service shown in the bundle authenticates by email/password, verifies Argon2id password hashes, and writes failed-login audit rows; I did not find a rate-limit or lockout check around failed admin login attempts.

Because `/admin/login` is now implemented, this is security-relevant and should be fixed before adding more admin management features.

**Fix:** add rate limiting by IP and normalized email. A DB-backed approach is acceptable if Redis is optional. At minimum:

```text
5-10 failed attempts per email/IP per window -> temporary lockout
audit both failed attempts and lockouts
return generic failure text
keep lockout metadata secret-free
```

## P1/P2 - Production provider-secret validation does not fully match configuration docs

The configuration doc says production requires strong provider secrets when OpenAI/OpenRouter providers are enabled. `slaif-api-gateway-concatenated.txt`

The settings tests visible in the bundle validate production HMAC/admin/one-time-secret requirements, placeholder detection, short-secret rejection, and one-time-secret key shape. They do not show equivalent production validation for `OPENAI_UPSTREAM_API_KEY` / `OPENROUTER_API_KEY` when the corresponding provider is enabled. `slaif-api-gateway-concatenated.txt`

This is not as severe as an accounting bug because provider adapter construction will fail later if a key is missing, but it is a docs/runtime contract mismatch and can cause production deployments to start but fail at request time.

**Fix:** when `APP_ENV=production` and a provider is enabled, require its configured upstream key env var to resolve at startup or readiness. Also warn or fail if `OPENAI_API_KEY` is present in the server environment and appears to be a real upstream provider key, because docs explicitly reserve that name for client gateway keys. `slaif-api-gateway-concatenated.txt`

---

## P2 - Admin role semantics are unclear

The admin model and CLI distinguish `superadmin` from `admin`. The admin CLI exposes `is_superadmin`/`role`, and `AdminService.create_admin_user()` stores either `superadmin` or `admin`.

But the admin route tests construct ordinary `role="admin"` users and exercise sensitive actions through session/CSRF validation.

That may be acceptable if all admins are intentionally full operators. But then the role distinction is misleading. If roles are intended to matter, provider config, pricing, FX, route management, usage repair, key rotation, and admin-user management should be permission-gated.

**Fix:** either document “all active admins are full operators for now,” or enforce role checks for sensitive operations.

---

## P2 - Email delivery is safe for plaintext handling, but exactly-once delivery semantics need hardening

The email delivery design is strong on secret containment: Celery tasks carry IDs only, and integration tests assert plaintext keys are not stored in email rows or audit metadata after sending.

The service intentionally locks and decrypts a one-time secret, attempts SMTP while the transaction remains open, and sets `consumed_at` only after SMTP success.

That policy prevents consumption on SMTP failure, but it leaves a classic edge case: SMTP can succeed and then the DB update/commit can fail, leaving the one-time secret retryable and possibly causing duplicate key email delivery.

**Fix before production email scale-up:** add an explicit delivery attempt state or outbox-style workflow that can distinguish “SMTP accepted but DB finalization failed” from ordinary failed delivery. This does not undermine the current no-plaintext-storage design, but it matters for operator trust.

# Required fixes before the next major feature-expansion round

I would fix these before expanding into more endpoints, bulk workflows, or richer dashboard functionality:

1. **Reject or properly account for `n > 1`.** This is the top hard-quota correctness risk.
2. **Fix `/v1/models` empty `allowed_models` behavior.** It must match documented policy semantics.
3. **Add admin login rate limiting / lockout.** The admin login surface is now real.
4. **Enforce production provider-secret requirements when providers are enabled.** Align runtime with configuration docs.
5. **Clarify or enforce admin roles.** Either all admins are full operators, or RBAC must be real.
6. **Update `docs/openai-compatibility.md` admin/email status.** Keep the contract docs synchronized.
7. **Harden email delivery exactly-once behavior.** Especially SMTP success followed by DB failure.

# Bottom line

Audit 5.0 shows a much more mature project. The previous central gateway blockers are mostly resolved: request passthrough, streaming usage, missing-usage semantics, provider-completed recovery, Redis long-stream concurrency, CLI secret safety, redaction, provider diagnostics, cost metrics, and DB readiness all look substantially better.

I would now describe the project as:

> **A serious pre-production OpenAI-compatible institutional gateway for Chat Completions, with strong quota/accounting architecture and credible operator tooling, but not yet production-release-ready.**

The main thing that still must not slip is `n > 1` accounting. After that, admin login rate limiting and the `/v1/models` policy mismatch are the next most important fixes before broad feature expansion.
