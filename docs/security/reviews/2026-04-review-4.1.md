# Outstanding blockers before feature expansion

## P0 blocker 1 — OpenAI request fields are still silently dropped

The request schema is still too narrow:

```python
class ChatCompletionRequest(BaseModel):
    model: str
    messages: list[dict[str, Any]]
    stream: bool | None = False
    max_tokens: int | None = None
    max_completion_tokens: int | None = None
```

Location:

- `app/slaif_gateway/schemas/openai.py:22-27`

The endpoint receives this model:

- `app/slaif_gateway/api/openai_compat.py:41-45`

Then the gateway forwards:

- `app/slaif_gateway/services/chat_completion_gateway.py:86`

Because Pydantic extras are not allowed/preserved here, ordinary OpenAI SDK fields are still dropped before reaching policy or provider forwarding.

This includes fields such as:

- `temperature`
- `top_p`
- `stop`
- `tools`
- `tool_choice`
- `response_format`
- `seed`
- `user`
- `presence_penalty`
- `frequency_penalty`
- `logprobs`
- `stream_options`

This is a **major compatibility blocker**. It means the gateway may accept a request, return a response, and yet silently alter the user’s intended model behavior.

### Required fix

Make the schema explicitly preserve extra fields, or parse the raw JSON body and validate only gateway-relevant fields.

For example:

```python
from pydantic import BaseModel, ConfigDict
from typing import Any

class ChatCompletionRequest(BaseModel):
    model_config = ConfigDict(extra="allow")

    model: str
    messages: list[dict[str, Any]]
    stream: bool | None = False
    max_tokens: int | None = None
    max_completion_tokens: int | None = None
```

Then add API-level tests, not just service-level tests:

1. POST `/v1/chat/completions` with `temperature`, `response_format`, `tools`, and `stream_options`.
2. Mock the upstream provider.
3. Assert the upstream body still contains those fields.

There is already a direct policy-service test showing arbitrary fields can be preserved if they reach the service. That is not enough because the API schema currently strips them first.

## P0 blocker 2 — Streaming still does not force final usage metadata

The streaming adapters still do this:

- `providers/openai.py:83-86`
- `providers/openrouter.py:84-87`

They copy the body, replace the model, and force `stream=True`. They do **not** inject or enforce a final-usage option.

That means the gateway’s accounting success path depends on the caller or provider somehow producing final usage metadata. But because the API schema currently drops `stream_options`, even a caller who tries to request usage metadata may not succeed through the public endpoint.

### Why this is critical

The streaming finalizer depends on `usage_chunk`:

- `chat_completion_gateway.py:254-255` — captures usage.
- `chat_completion_gateway.py:262-282` — finalizes success only when `completed and usage_chunk`.
- `chat_completion_gateway.py:283-302` — completed stream without usage is recorded as incomplete/failure and releases the reservation.

So the system has a strong accounting requirement, but the adapter does not enforce the provider behavior needed to satisfy that requirement.

### Required fix

For streaming calls, inject usage-request metadata in the provider adapter before sending upstream.

Conceptually:

```python
if body.get("stream") is True:
    stream_options = dict(body.get("stream_options") or {})
    stream_options["include_usage"] = True
    body["stream_options"] = stream_options
```

Then add tests that assert the exact forwarded provider body contains the usage option.

## P0 blocker 3 — Redis concurrency is still not true active concurrency

The Redis concurrency logic still removes active entries based on the rate-limit window:

- `rate_limit_service.py:38-40`
- `rate_limit_service.py:67-70`
- `rate_limit_service.py:177-196`

The script removes entries older than:

```text
now_ms - window_ms
```

That means a stream that lasts longer than the configured rate-limit window can disappear from the active concurrency set while still running.

### Example failure

Suppose:

- concurrency limit = 2,
- window = 60 seconds,
- two users start long streaming responses,
- both responses last 5 minutes.

After about 60 seconds, those entries can be cleaned from Redis even though both streams are still active. More streams can then be admitted, violating the configured concurrency limit.

### Required fix

Separate request-rate windows from active concurrency.

A production-safe model should use:

- request/minute counter,
- estimated-token/minute counter,
- active-concurrency set/hash independent of request window,
- long active-slot TTL,
- explicit release by request ID,
- heartbeat/refresh for streaming requests,
- stale cleanup only after a conservative max-active duration.

Add a test where:

1. concurrency limit is 1,
2. rate window is short,
3. first stream remains open past the window,
4. second request is still rejected until the first stream releases.

## P0 blocker 4 — No durable recovery state for provider-success/accounting-failure

The code now handles provider errors and missing usage better. But it still does not solve the hardest streaming accounting failure:

> The provider successfully generated and streamed content, but database accounting finalization fails after content was already sent.

The current success finalization path:

- `chat_completion_gateway.py:262-282`
- `chat_completion_gateway.py:612-622`
- `accounting.py:174-259`

The system yields content chunks before finalization, then tries to finalize accounting. If finalization fails, the client may already have received the model output.

Current reservation statuses appear limited to:

- `pending`
- `finalized`
- `released`
- `expired`

Location:

- `db/models.py:29-39`

There is no explicit durable state such as:

- `finalization_failed`
- `provider_succeeded_accounting_failed`
- `needs_reconciliation`

### Why this matters

A production accounting system needs a durable record that says:

> The provider call likely succeeded, the user received output, but accounting failed and must be reconciled.

Without that, this failure can become either:

- an unrecoverable pending reservation,
- a released reservation with undercounted cost,
- a client-visible stream error after useful content was already delivered,
- or an operator mystery.

### Required fix

Add a durable recovery path.

Recommended approach:

1. Add reservation or ledger status for `finalization_failed` / `needs_reconciliation`.
2. Capture provider usage metadata before trying to finalize counters.
3. If ledger/counter finalization fails, persist a minimal durable recovery record if possible.
4. Add a reconciliation command that can finalize or safely mark these cases.
5. Add tests where content is streamed, usage is received, and accounting finalization fails.

This is different from stale pending reservation cleanup. The updated reconciliation service is good, but it does not address this provider-success/accounting-failure class.

## P0 blocker 5 — CLI JSON output still exposes plaintext keys

The key CLI still includes plaintext secrets in JSON output:

- `cli/keys.py:258-269`
- `cli/keys.py:272-283`
- `cli/keys.py:585-588`
- `cli/keys.py:1075-1078`

The tests also explicitly assert this behavior:

- `tests/unit/test_cli_keys_create.py`
- `tests/unit/test_cli_keys_rotation.py`

This is not safe enough for a production operator CLI. JSON output is frequently captured by CI systems, shell wrappers, terminal logs, command transcripts, and support tickets.

### Required fix

Change the default:

- no plaintext secret in JSON by default,
- no plaintext secret in structured output unless explicitly requested,
- add `--print-secret` or similar if absolutely needed,
- preferably add `--secret-output-file PATH`,
- create secret-output files with restrictive permissions,
- print warnings to stderr, not JSON payloads.

A better operator pattern:

```bash
slaif keys create ... --secret-output-file ./gateway-key.txt
chmod 600 ./gateway-key.txt
```

or:

```bash
slaif keys create ... --print-secret
```

with loud warnings.

The current “plaintext shown only once” principle is good, but emitting it into JSON by default weakens that principle.

# Important P1 issues

## 1. Redaction still does not fully handle custom key prefixes

The redaction regex is still hard-coded around selected prefixes:

- `utils/redaction.py:23-26`

It covers things like `sk-slaif`, `sk-ulfe`, `sk-or`, `sk-proj`, and `sk-test`, but the configuration supports custom accepted prefixes. A custom gateway key prefix may not be redacted in free-form logs unless it appears inside a `Bearer ...` pattern.

### Required fix

Build redaction patterns from configured accepted prefixes, or use a generic gateway-key shape based on the public-id/secret format.

Also add tests for a configured custom prefix such as:

```text
sk-acme-prod-...
```

---

## 2. Gateway key redaction still reveals secret fragments

`utils/crypto.py:84-100` still redacts a gateway key by showing the first and last four characters of the secret fragment.

For bearer credentials, showing any secret characters is unnecessary. The public key ID is already enough for support/debugging.

Recommended output:

```text
sk-slaif-<public_id>.<redacted>
```

not:

```text
sk-slaif-<public_id>.abcd...wxyz
```

---

## 3. Accounting metadata sanitizer is too exact-match based

`accounting.py:575-583` filters forbidden metadata keys by exact lower-case key matching. That catches obvious names like `api_key`, but not necessarily variants such as:

- `apiKey`
- `providerApiKey`
- `authorizationHeader`
- `openaiKey`
- `secretValue`
- nested sensitive objects with non-standard names.

The general redactor is stronger than this, but accounting metadata needs its own robust sanitizer because ledger metadata is durable.

### Required fix

Normalize keys before matching:

- lower-case,
- remove `_`, `-`, spaces,
- check substrings like `apikey`, `authorization`, `bearer`, `secret`, `token`, `password`.

Also recurse through nested mappings/lists and apply redaction to string values.

## 4. Cost metric exists but is not recorded

The metric function exists:

- `metrics.py:147-151`

But the chat completion success path appears to record tokens only:

- `chat_completion_gateway.py:672-698`

The cost metric should be incremented after successful accounting finalization. This may require returning the finalized accounting result from `_finalize_successful_chat_completion()` rather than discarding it.

### Required fix

Record:

- provider,
- upstream model,
- endpoint,
- billed cost EUR.

Add a test that a successful non-streaming and streaming call increments cost metrics.

---

## 5. Redis release failures need their own metric

`_release_rate_limit_concurrency()` currently records release failures as rate-limit rejections:

- `chat_completion_gateway.py:423-438`

That is misleading. A failed release is operationally different from a request rejected by the limiter.

### Required fix

Add dedicated metrics such as:

- `gateway_rate_limit_release_failures_total`
- `gateway_rate_limit_concurrency_slots_active`
- `gateway_rate_limit_concurrency_stale_cleanups_total`

---

## 6. Missing-usage stream currently ends with `[DONE]`

In the completed-without-usage path, the code records the request as incomplete/failure, releases the reservation, sets provider status to incomplete, and then yields the final done event:

- `chat_completion_gateway.py:283-302`

This creates a confusing client/operator mismatch:

- client may see a normal completed stream,
- accounting records a failed/incomplete zero-actual-cost ledger entry.

That may be acceptable as a temporary compatibility compromise, but it should be an explicit documented policy. For production accounting, I would prefer either:

1. emit an error event instead of a clean `[DONE]`, or  
2. treat this as a reconciliable provider-success/missing-usage state, not ordinary failure.

---

## 7. DB engine settings are still basic

The app lifespan/session handling is good:

- `lifespan.py:18-41`
- `db/session.py`

But production DB settings should expose:

- pool size,
- max overflow,
- pool recycle,
- connection timeout,
- statement timeout,
- `pool_pre_ping`.

This is not as urgent as the streaming/accounting issues, but it matters before real deployment.

---

## 8. `/readyz` details should be internal-only

`api/health.py` exposes DB and Alembic readiness details. This is useful for operators, but in production it should not be publicly reachable.

This can be solved at deployment/ingress level, but the docs should be explicit:

- `/healthz` can be public-ish,
- `/readyz` should be internal,
- `/metrics` should remain allowlisted/authenticated in production.

Grade: **C+**

- provider errors,
- completed streams without usage,
- Redis concurrency release,

# Final assessment

The updated project is clearly moving in the right direction. The reconciliation work and expanded streaming tests are real improvements, not superficial changes. The codebase still reads like a serious infrastructure project rather than a simple key proxy.

However, the highest-priority previous recommendations were only partially followed. The project still has three central mismatches with its own intended positioning:

1. It claims OpenAI compatibility, but still drops many normal OpenAI request fields.
2. It depends on final streaming usage for accounting, but does not force that usage to be returned.
3. It exposes a concurrency limit, but long streams can age out of Redis while still active.

Fixing those would move the project from **B- / credible alpha** toward **B+ / serious pre-production candidate**.
