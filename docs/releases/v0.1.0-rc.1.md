# SLAIF API Gateway v0.1.0-rc.1

## Status

RC-beta / release-candidate beta for the implemented and documented scope.

This release candidate is not production certified, is not a compliance
attestation, and is not a penetration-test report.

## What This Release Is

SLAIF API Gateway is an OpenAI-compatible gateway for educational and
institutional LLM access. It issues gateway-owned API keys, enforces local
policy and hard PostgreSQL-backed quota/accounting, forwards permitted requests
to OpenAI or OpenRouter, and provides an admin dashboard plus CLI for operators.

Users keep using standard OpenAI-compatible client configuration:
`OPENAI_API_KEY` is the gateway-issued key and `OPENAI_BASE_URL` points at the
gateway `/v1` base URL.

## Implemented API Scope

- `GET /v1/models`.
- `POST /v1/chat/completions` non-streaming.
- `POST /v1/chat/completions` streaming with SSE.
- OpenAI-shaped errors for unsupported `/v1` endpoints and policy failures.
- `n` omitted or `n=1` is allowed; `n > 1` is rejected before provider
  forwarding, rate limiting, pricing, quota reservation, or ledger mutation.
- Streaming requests force `stream_options.include_usage=true`.
- Missing final streaming usage emits a safe stream error rather than a
  misleading successful `[DONE]`.
- `/v1/models` visibility follows the authenticated key's allow-list policy.
- Client `Authorization` headers are never forwarded upstream; the gateway
  substitutes server-side provider credentials.

Unsupported endpoints, including Responses, embeddings, files, images, audio,
and native Anthropic APIs, remain documented limitations.

## Implemented Admin Dashboard Scope

- Admin authentication with server-side sessions, CSRF protection, and login
  rate limiting.
- Active admins are full operators. `superadmin` is metadata/future-proofing and
  is not an enforced RBAC boundary in this release candidate.
- Key list/detail/create, rotate, suspend, activate, revoke, validity update,
  hard quota update, usage-counter reset, and explicit key email delivery modes.
- Bulk key import preview and confirmed execution for `none`, `pending`, and
  `enqueue` email modes. Bulk synchronous `send-now` is not implemented and
  rejects before mutation.
- Owner, institution, and cohort list/detail/create/edit metadata workflows.
- Provider config, model route, pricing, and FX metadata management.
- Route, pricing, and FX import preview/execution workflows.
- Usage, audit, and email delivery views.
- Usage and audit CSV metadata exports with CSRF, confirmation, audit reason,
  row limits, formula-injection mitigation, and secret/content exclusion.

## Implemented CLI, Deployment, And Ops Scope

- Typer CLI commands for admin bootstrap, key lifecycle, owner/institution/cohort
  records, provider/route/pricing/FX metadata, usage reports/exports,
  reconciliation, email delivery helpers, and database migration helpers.
- Docker Compose services for API, worker, scheduler, PostgreSQL, Redis, and
  Mailpit.
- Explicit migrations through `slaif-gateway db upgrade`; API, worker, and
  scheduler containers do not run migrations automatically.
- Nginx example configuration with streaming-safe proxy settings.
- Health and readiness endpoints, metrics, structured redaction-oriented
  logging, scheduled reconciliation inspection/execution controls, and optional
  reconciliation alerts.
- GitHub Actions CI for unit/lint, PostgreSQL integration, OpenAI-compatible
  E2E, Playwright browser smoke, Docker Compose smoke, Nginx validation, docs
  hygiene, CodeQL, and Dependabot.

## Security Posture

- Gateway keys are stored as HMAC digests only.
- One-time key delivery secrets are encrypted.
- Provider keys are referenced by environment variable names and are not stored
  in provider config rows.
- Client `Authorization` headers are not forwarded to upstream providers.
- Celery key email tasks carry IDs only, not plaintext gateway keys.
- Plaintext gateway keys are shown only once on explicit no-cache result pages
  or delivered through the one-time-secret email path.
- Bulk `enqueue` suppresses browser plaintext display.
- Prompt and completion content is not stored by default.
- Admin state-changing forms use CSRF protection, and high-risk/mutating
  workflows require confirmation and/or an audit reason according to the action.
- Production provider-secret validation rejects missing or placeholder upstream
  provider secrets for enabled production providers and protects the
  `OPENAI_API_KEY` client/server boundary.

## Known Limitations

- Bulk synchronous `send-now` key import execution is not implemented.
- Native Anthropic API is not implemented; Anthropic-family models can be routed
  through OpenRouter's OpenAI-compatible interface.
- Responses API is not implemented.
- Embeddings API is not implemented.
- File, image, audio, and batch endpoints are not implemented.
- MFA is not implemented.
- Full RBAC is not implemented; every active admin is a full operator.
- Owner/institution/cohort delete and anonymization workflows are not
  implemented.
- External FX refresh workflows are not implemented.
- Arbitrary old plaintext key resend is intentionally not implemented.
- Real upstream smoke tests are disabled by default and require explicit
  operator opt-in plus real provider credentials.
- Production deployment still requires operator-managed secrets, HTTPS/Nginx
  hardening, backups, monitoring, alert routing, and operational runbooks.
- There is no formal security certification.
- There is no formal penetration test.
- There is no SOC, ISO, or other compliance attestation.

## Upgrade / Install Notes

For first-time local use, follow [`../quickstart.md`](../quickstart.md).

Compact local Docker sequence:

```bash
git clone https://github.com/ulfe-lmi/slaif-api-gateway.git
cd slaif-api-gateway
cp .env.example .env
docker compose build
docker compose up -d postgres redis mailpit
docker compose run --rm api slaif-gateway db upgrade
docker compose up -d api worker scheduler
```

Then create an admin account and local metadata as described in the quickstart.
Replace placeholder secrets before any real deployment.
